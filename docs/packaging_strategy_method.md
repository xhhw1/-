# 包装策略 Agent 方法论

包装策略 Agent 固定按以下链路输出方案：

**核心卖点 → 画面事件 → 产品证据 → 文案层级 → 信息标识 → 出图约束**

## 方法说明

1. 核心卖点  
   先判断哪条卖点最适合成为包装正面的主视觉叙事，其他卖点分配到辅助图、功能图标、侧面或背面。

2. 画面事件  
   把卖点变成能被看见的动作、状态或变化，例如拼装前后、触碰反应、发光变化、系列并排、配件展开、局部特写。

3. 产品证据  
   为每个画面事件指定证据来源或参考图类型，例如产品成品图、爆炸图、玩法步骤图、配件图、系列图、LOGO/VI 图、竞品图。

4. 文案层级  
   正面主图只保留一处卖点文字表达区，通常是一句主标题或一句主卖点短文案。不要同时在副标题、功能图标、底部信息条、卖点徽章里重复表达卖点。

5. 信息标识  
   正面只保留必要品牌/品名/年龄/系列识别，不在正面底部生成合规认证区、安全提示区、条码区、厂商信息区。这些内容进入背面策略。

6. 出图约束  
   区分硬约束和创意方向。产品形态、颜色、结构、配件数量、LOGO 使用是硬约束；背景氛围、辅助图摆放、模特/手部、图标数量是创意方向。

## 品类知识调用

包装策略 Agent 需要在输出方案前调取相关品类知识。品类知识不写死在后端代码里，而是作为知识库条目存放在：

- 默认种子文件：`src/ai_visual_agent/knowledge/defaults/*.json`
- 运行时存储：`knowledge_entries` 表或内存 Store
- 管理入口：前端「知识库」模块和 `/api/knowledge` API

Agent 调用规则：

- 根据项目品类、用户描述、已确认卖点和工作流类型自动检索知识库。
- 命中的知识只能作为设计方法和表达原则，不能覆盖产品资料、用户确认内容和 VI 事实。
- 如果知识与产品事实冲突，优先相信产品事实，并写入 `risk_notes`。

默认随系统提供一条「娱乐/玩具类线下包装知识」作为种子条目。该条目可以在知识库模块中编辑、停用、删除或替换。

该默认条目的策略重点：

- 3 秒货架识别：孩子先被主题/角色/玩法吸引，父母快速确认品类、年龄、价值和安全感。
- 双受众表达：孩子看酷、好玩、想拥有；父母看动手能力、益智价值、品质可信、信息清楚。
- 产品可见性优先：通过开窗、产品大图、局部特写或比例参照降低想象成本。
- 玩法证据前置：用步骤图、手部动作、前后变化、功能反馈图或小场景证明核心玩法。
- 系列感驱动复购：多 SKU 时用编号、系列小卡、角色阵列或收集徽章建立持续购买动机。

娱乐/玩具类首选设计风格：

- 世界观主视觉型：适合角色、IP、科幻、奇幻、宠物、探索、变形类产品。
- 玩法证据型：适合拼装、互动、实验、机关、步骤或前后变化类产品。
- 系列收藏型：适合多 SKU、角色编号、隐藏款、可收集体系。
- 礼赠开窗型：适合实物颜值、材质、尺寸或成品状态是重要购买理由的产品。

Badge 设计：

- 1 个主徽章承载最强购买理由。
- 2-3 个辅助徽章承载功能、年龄、系列、配件、玩法证明。
- 正面核心徽章建议 3 个以内，最多不超过 5 个。
- 每个徽章文案尽量 2-8 个中文字符。
- 徽章颜色应来自 VI 色或产品高频色，避免每个徽章一个颜色导致廉价。

## 好策略的判断标准

- 不是「主视觉居中 + 科技背景 + 突出卖点」这种空泛描述。
- 是「某条卖点通过某个画面事件被看见，并由某类产品证据支撑」。
- 辅助图不是装饰，而是证明卖点成立。
- 文案、图标、辅助图和信息区草案都服务卖点证明与购买确认。
- 策略给清楚方向和边界，但不把所有构图细节锁死成唯一排版。

## 前端审核展示

包装策略在后端仍保存为 `PackagingStrategy`，用于人工编辑、版本记录、下游 Agent 调度和提示词转换；但在前端普通审核态不展示为 JSON 或字段表格，而是展示为「正面包装构图详解」报告。

报告默认包含：

- 产品核心图展示：主视觉如何抓住第一核心卖点。
- 辅助产品图展示：次要卖点由哪些参考图、示意图或小图证明。
- 背景图与背景色：世界观、光影、地台、纹理与 VI 的关系。
- 模特或手部融入方式：只有服务互动、尺寸或情绪时出现。
- 主卖点文案：唯一一处正面卖点表达区。
- 辅助标识：LOGO、品名、年龄、系列编号等必要识别；正面不放合规认证区、安全提示区、条码区或厂商信息区。

用户需要修改时，可以进入编辑模式修改结构化字段；确认后，系统把报告式策略交给「主图提示词 Agent」转换成 GPT-Image-2 图生图提示词。
